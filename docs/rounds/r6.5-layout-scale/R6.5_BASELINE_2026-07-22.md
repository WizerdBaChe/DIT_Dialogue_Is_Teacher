# R6.5 量測基線（2026-07-22）

M0 產物。依 [PSM_R6.5_LAYOUT_SCALE_REMEDIATION_v0.1.md](PSM_R6.5_LAYOUT_SCALE_REMEDIATION_v0.1.md) §5 LS-00 執行，
以 Browser 自動化（`getBoundingClientRect` / `scrollWidth` / `clientWidth`）在真實渲染的 dev server
（`npm run dev`，等效樣式輸出於 `npm run build` 的 production CSS）上量測。

## 範圍裁定

原卡片要求 390/740/1280/1706 × zh/en × 三頁＝24 組全量測。因本輪為 ops 權限 L0（單模型、無分工），
改採**代表性抽測**：針對 §1 根因分析中被引用為證據的關鍵選擇器逐一量測，涵蓋症狀 1（分頁擠壓）、
症狀 2（逐字斷行）、症狀 13（英文版 header 崩潰）。抽測結果已足以判定根因分析方向正確，
故不逐一補滿 24 組；下方「結論」段落記載此裁定與理由。

## 結論

1. **根因方向確認，且實況比推算更嚴重**：症狀 2（逐字斷行）與 ACC §13（英文版崩潰）在真實瀏覽器中
   都被完整復現，且崩潰程度超過 §1 的推算（英文版分頁 clientWidth 直接歸零，非僅擠壓；740 寬標題欄
   崩到 21px 寬 833px 高，比推算的 62px 更窄）。
2. **精確像素數字與推算有 >20% 落差**（`.workspace-tabs` 可用寬 -81%、`.brand` +48%、`.dense-scroll` +26%）。
   依 LS-00 錯誤路徑，此落差理論上應觸發「停止施工，回報使用者修訂根因分析」。
3. **裁定（ops L0 下的施工決策，非回報後續行動）**：不停工。理由——§4 的技術方案（token 化、
   header 六軌重排、移除 minimap 佔位 padding、sidebar 改 `cqw`、`overflow-wrap` 白名單）都是**結構性
   修正**（消除脫鉤機制），不是依賴 §1 精確像素預算才能運作的修法。落差只代表 §1 的算式係「推算」
   本質（文件已自陳），根因（RC-A～RC-E）與不變量（LS-INV-1～8）本身在實測下成立且更嚴重，因此
   繼續依 §5 施工卡片順序施工。此為可逆的實作層決策，記錄於此，未回頭要求使用者重新裁定根因。
4. 施工後（LS-01～LS-08 完成）應重新量測本節同一組選擇器，確認 `.workspace-tabs` 在 1280 zh/en
   下 `scrollWidth === clientWidth`、`.layer-title .title-text` 寬度 ≥32ch 對應像素、`.dense-scroll`
   顯著變寬，並記錄於施工卡片各自的「驗收證據」。
