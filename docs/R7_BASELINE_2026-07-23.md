## 5. 模擬 ×1.5（A4.4 宣告）：`.header` 高度與 `.workspace-tabs` 溢位

模擬 CSS（與 PSM §A4.4 逐條一致）：
```css
:root { --chrome-scale: 1.5; }
.header { padding: 4px 18px; }
.brand h1 { font-size: calc(17px * var(--chrome-scale)); line-height: 1.2; }
.workspace-tab { font-size: calc(12.5px * var(--chrome-scale)); min-height: 40px; }
.header .btn { font-size: calc(12.5px * var(--chrome-scale)); }
.compact-action { width: calc(30px * var(--chrome-scale)); }
.compact-replay { min-width: calc(54px * var(--chrome-scale)); }
```

| 寬度 | 語言 | `.header` 高度（×1.5） | `.workspace-tabs` scrollWidth / clientWidth | 判定 |
|---|---|---|---|---|
| 390 | zh | 114 | 354 / 354 | 雙列既有基線本就 >56，不適用單列判定 |
| 390 | en | — | — | 未逐項量測（雙列基線已知不適用56px單列判定，優先度低） |
| **740** | **en** | **68** | **323 / 231（溢位 92px）** | ❌ **超過 56px，且 tabs 溢位** |
| 740 | zh | 56 | 215 / 215 | OK |
| 1280 | zh | 56 | 215 / 215 | OK |
| 1280 | en | 56 | 323 / 323 | OK |
| 1706 | zh | 56 | 215 / 215 | OK |
| 1706 | en | 56 | 323 / 323 | OK |

**根因**：A4.4 推算「1280 English 為最緊情境」只考慮了寬度足夠時的水平擠壓，沒有考慮到
**740 是 `@container dit-app` 從雙列切回單列的邊界寬度（`max-width: 719px` 之上即單列）**——
740 剛好是單列 header 在「最窄可用寬度」下疊加「English 最長 tab 文字」疊加「×1.5 放大」的三重最壞情境，
比 1280 English 更緊。1280 因為有更多水平空間吸收放大後的字級，反而沒事；740 的水平餘裕本來就少，
放大 1.5 倍後 `.workspace-tabs`（`flex: 0 0 auto` 不可壓縮，LS-INV-2）直接把可用空間吃穿，
連帶把 `min-height: 40px` 的 tab 撐出兩列，header 總高度因此超過 56px。

## 待使用者裁定的問題

1. **D-R7-03 二選一**：740 English 情境下，「header 56px 硬上限」與「chrome-scale ×1.5」兩者
   無法同時成立（68px vs 56px，且 tabs 溢位 92px）。R7A-05 不會自行調降倍率或放寬高度——
   需要使用者決定：(a) 降低倍率（例如 ×1.35，需重新量測是否夠）、(b) 740 用不同於 1280 的
   `--chrome-scale` 值（雙旋鈕分層）、(c) 允許 740 情境下 header 換成雙列（比照 390 既有設計，
   但 740 目前是刻意單列，改雙列是設計變更）、(d) 其他方案。
2. **R7-INV-4 措辭疑義**：不變量寫「`.header` min-height 56px 為上限亦為下限」，但 390 寬度下
   現行（已驗收）header 本就是 86px（雙列）。建議把不變量改為「**單列情境**（`@container dit-app`
   ≥720px）下 `.header` 高度上限亦為下限 56px；雙列情境維持既有 R6.5 基線，不在本次 56px 約束範圍」，
   但這是措辭層級的決定，需使用者確認後才能動 PSM 文件。
