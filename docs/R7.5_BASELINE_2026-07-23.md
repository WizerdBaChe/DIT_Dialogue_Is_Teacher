## 施工後追加發現：RC-6／Auto-review 審查子代理噪音（W8）

M1（W1/W2/W6）施工完成、UAT 進行中，你回報 `rollout-2026-07-23T04-53-49-019f8b9b-13f9-7430-8d74-6b4659458069.jsonl` 等 session 的第一張卡是一段完整的「Codex agent history whose request action you are assessing...」轉述文字。追查後確認：這是 OpenAI Codex CLI 官方文件化的 **Auto-review（Guardian Approval）** 功能——一個獨立審查子代理（`codex-auto-review` 模型）在動作被執行前，把「先前完整歷史」轉述給自己評估，吐出 `{"outcome":"allow"/"deny"/...}` 結構化裁決，整段審查對話與主線對話記錄在同一個 rollout 檔案裡。

**與 RC-1／RC-2 都不同的第三種噪音**：不是 XML 標籤前言（INV-3 白名單抓不到），也不是零內容 metadata（有真實的 allow/deny 裁決結果）。對 21 個樣本重新掃描：**174 則審查轉述 + 47 則裁決，分布在 11 個檔案**；其中原本在上方步驟 3 被當「乾淨對照組」的 `…56-5167…jsonl`，84 張 user_text 卡裡 **83 張其實是這種噪音**（已更正對照組，見上）。

**處置（W8，你核准的方向：一次性彈窗告知 + 逐則精簡標記卡）**：
- `src/core/adapters/codexJsonl.ts` 新增偵測：轉述文字固定以 `"The following is the Codex agent history"` 開頭；緊接著的下一則 assistant 回覆若是短 JSON `{"outcome":"..."}` 即為裁決，兩者各自精簡成一則自我解釋標記卡（沿用既有 `turn_aborted` 慣例），裁決結果原樣顯示（含未知代碼時退回顯示原始字串，不猜測）。
- `finish()` 新增聚合 warning，經既有 `warnings` → 可關閉 `NoticeBanner` 機制一次性顯示，符合「彈窗告知」且未新增 UI 機制。
- 對照組檔案 `…56-5167…jsonl` 實測：處置前 user_text=84（83 假），處置後 user_text=0、assistant_text=15（真實回覆）、unknown（含審查標記卡）=151，聚合 warning 正確顯示筆數。

此發現與處置超出原 PSM R7.5 核准範圍，屬追加工作項（W8），已取得你的明確核准後施工，測試詳見 `codexJsonl.test.ts`。
