# Concept Note — 概念演化軌跡（暫名 CET, Concept Evolution Tracker）v0.2

> 日期：2026-07-04｜狀態：**構想存檔，未立項**。DIT 剩餘藍圖（R1→R7→…）優先，本產品不插隊。
> 本文件為 why 層（Concept Note）；若拍板啟動，下一步是它自己的 RPD，不是直接寫碼。

## 1. 一句話與不可化約的痛

**痛**：長篇 AI 設計/腦暴對話結束後，「概念是怎麼被提出、質疑、精煉、最終收斂成架構」的建構過程
埋在線性對話裡，無法複盤、無法學習——只留下結論，丟失了推理軌跡。

**產品**：把 AI 對話（Markdown/文字）蒸餾成「概念依賴樹」——
`概念提出 → 被質疑 → 精煉 v2 → 與其他概念建立關係 → 收斂為架構層級`，
支線掛反例/被否決的方案，節點可 drill-down 回原文段落。

**第一個真實使用者**：使用者本人，複盤自己的產品設計對話（如 DIT 的 RPD 討論 session）。

## 2. 先行研究（2026-07-04 掃描）

| 前例 | 是什麼 | 與本構想的差距 |
|---|---|---|
| IBIS / gIBIS（1960s–1988, Kunz & Rittel）、Dialogue Mapping（Conklin） | 「議題→立場→論證」的正典方法論，wicked problem 分析 | **人工即時建圖**；本構想是事後自動蒸餾 |
| Compendium（Open University） | IBIS 標記法的概念建圖工具，事實標準 | 同上，手動；且無「概念版本演化」時間性 |
| Kialo | 樹狀正反論證平台 | 眾人辯論導向，非個人對話複盤 |
| Neo4j LLM Knowledge Graph Builder 等 LLM→KG 工具 | 非結構文本→實體關係圖 | 產出是**靜態實體圖**，不是概念隨對話輪次的**演化軌跡**；重基礎設施（圖資料庫） |

**結論**：(1) 差異化成立——「事後自動 + 演化時間性 + 溯源 drill-down + 本地優先」的組合無人佔位；
(2) **IBIS 語彙是現成的節點分類學**（Question/Idea/Pro/Con），蒸餾 preset 不必發明語彙，
建議映射為：`question(質疑) / idea(概念提出) / refine(精煉) / relate(建立關係) / converge(收斂定案)`
＋支線 `counter-example(反例) / rejected(被否決方案)`。

## 3. 工程難點——對「只需加 Adapter + 新降噪規則」的誠實修正

架構同構判斷**七成正確**：管線形狀（Adapter → Normalize → Distill → ViewModel）、魚骨視圖、
drill-down、Provider 層、容錯哲學全部可沿用。但有三處被低估，正是本構想的真正工程量：

1. **Distill 從確定性變語意判斷（最關鍵）**。DIT 的降噪能走純規則，是因為 `.jsonl` 自帶結構
   （tool_use/isError 是機器可判的）。「這段話是在精煉概念 A」是語意判斷——確定性規則只能給
   弱基線（分輪、標題切段、關鍵詞重疊），真正的概念辨識/同指涉（同一概念換了說法）/精煉偵測
   **需要 LLM 進管線核心**。這顛覆 DIT「LLM 完全可選、純前端零依賴可用」的業務規則——
   CET 的核心價值本質上依賴 LLM（本地 Ollama 可，但不可無）。
2. **契約語彙不同**。`SpanType`（tool_use…）與 `SkeletonNode/RibKind`（objective/error/retry…）
   是任務執行語意；概念演化需要平行的新 preset（§2 的 IBIS 映射），且 DIT skeleton v1 已凍結
   （PSM ADR-010）——是「加一套新契約」，不是「調參數」。
3. **輸入與概念同一性**。「AI 對話 Markdown」來源不一（claude.ai 匯出/複製貼上/甚至同一份
   `.jsonl` 只取對話層）；parsing 容易，**跨數十輪追蹤「概念 A 被改名/被合併」才是難題**，
   需要在 Distill 層設計概念註冊表（concept registry）與版本鏈，這在 DIT 中沒有對應物。

## 4. 定位建議：開新系列產品，共用「架構模式」而非強行共用程式碼

| 選項 | 評估 |
|---|---|
| (A) 做成 DIT 新模式 | ❌ 稀釋 DIT「coding transcript 教學」定位；打破「LLM 可選」承諾；契約被迫混入兩套語意 |
| (B) **新系列產品（Lobby sibling），複製 DIT core 起步** | ✅ 建議。DIT 架構已被驗證，照樣式起新專案成本低；兩產品各自保持定位純粹 |
| (C) 先抽共用核心庫再各自消費 | ❌ 過早抽象——第二個消費者都還沒長出來就抽介面，抽錯的機率高於抽對 |

B 的附帶決策：待 CET 真的立項且長出形狀後，再回頭評估抽共用核心（那時有兩個真實消費者，
介面切在哪裡有證據可依）。這符合「可替換弱點、不為擴充而擴充」。

## 5. MVP 輪廓（供未來 RPD 起點，非承諾）

貼上/上傳對話 Markdown → 確定性分輪切段 → LLM 蒸餾（Ollama，概念抽取＋演化分類，IBIS 語彙）
→ 概念演化魚骨（與 DIT 同構的視圖語言）→ drill-down 回原文段落（溯源）。
驗證指標：使用者對一份自己的真實設計對話，能在 2 分鐘內講出「哪三個概念是怎麼收斂的」。

## 6. 關鍵決策點（另開討論時的議程，依影響大小排序）

| # | 決策點 | 選項與傾向 | 為什麼關鍵 |
|---|---|---|---|
| K1 | **LLM 進管線核心的接受度** | 傾向：接受「無 Ollama 時降級為弱基線分段視圖」 | 決定產品承諾（隱私/離線）與 MVP 範圍，是與 DIT 最根本的分歧點 |
| K2 | **概念同一性策略** | (a) 純 LLM 概念註冊表；(b) LLM 初判＋使用者手動合併/改名輔助（傾向 b，人機協作可自我修正） | §3 第 3 點的難題；做錯則整棵樹不可信 |
| K3 | **第一個輸入 adapter** | claude.ai 匯出 / 複製貼上 Markdown / Claude Code `.jsonl` 對話層（傾向：複製貼上最通用先行） | 決定 fixture 與解析工程量 |
| K4 | **視圖形態（UX 語意，須使用者拍板）** | (a) 沿用魚骨同構（開發最省）；(b) 概念依賴樹/DAG（更忠實於「依賴」語意但需新視圖引擎） | 概念可多父（依賴多個前概念），魚骨是線性主線——同構假設在這裡最可能破 |
| K5 | **蒸餾語彙定稿** | 傾向：IBIS 映射 `idea/question/refine/relate/converge` ＋支線 `counter-example/rejected` | 契約一旦定稿即凍結（沿用 DIT 的 schema 演進規則） |
| K6 | **命名與定位** | CET 為暫名；Lobby 系列 sibling（已傾向，見 §4） | 影響 repo 與文件鏈起點 |
| K7 | **共用核心抽取時機** | 已傾向：延後至 CET 成形後再評估（§4 選項 C 的理由） | 避免過早抽象 |

## 7. 啟動前置（硬條件）

1. K1–K6 於立項討論中拍板（K7 可留）。
2. 使用者提供 **3 份真實對話樣本**作 fixture——沒有真樣本不啟動（DIT 的經驗：fixture 先行）。
3. 走完整文件鏈：本 Concept Note → CET 自己的 RPD →（拍板後）PSM，才進實作。
